# Error Model

Document

- ID: error-model
- Version: 1.0
- Status: Stable

Purpose

- Define canonical runtime error model
- Define error categories
- Define error propagation
- Define error ownership across Sovrunn Runtime

Definition

Error Model defines the canonical structure used by Sovrunn Runtime to represent failures.

An Error describes why semantic intent could not be validated, planned, executed, or returned successfully.

Principles

MUST

- Be deterministic
- Be protocol independent
- Be engine independent
- Preserve failure cause
- Preserve semantic integrity
- Support safe error propagation
- Separate error details from successful results

MUST NOT

- Hide failure
- Convert failure into success
- Expose unsafe internal details
- Expose raw downstream engine errors directly to clients
- Modify SIR
- Modify Execution Plan

Error Categories

Error

MUST belong to one category.

Categories

- Protocol Error
- SIR Validation Error
- Planning Error
- Capability Error
- Execution Error
- Transaction Error
- Session Error
- Security Error
- Policy Error
- Timeout Error
- Cancellation Error
- Downstream Engine Error
- Runtime Error

Error Model

Error

MUST contain

- Error Identifier
- Error Code
- Error Category
- Error Message
- Error Severity
- Error Source
- Error State
- Retry Classification
- Trace Identifier
- Timestamp

MAY contain

- Execution Identifier
- Request Identifier
- Session Identifier
- Transaction Identifier
- Operation Identifier
- Resource Reference
- Capability Reference
- Engine Reference
- Cause
- Safe Details
- Internal Details
- Remediation

Error Severity

Severity

MUST be one of

- Info
- Warning
- Error
- Fatal

Error State

State

MUST be one of

- Detected
- Propagated
- Handled
- Retried
- Failed
- Suppressed

Suppressed

MUST only be used for explicitly handled non-fatal errors.

Retry Classification

Retry Classification

MUST be one of

- Retryable
- Non Retryable
- Retry With Backoff
- Retry After Remediation
- Unknown

Timestamp

Timestamp

MUST

- Use adopted time representation standard
- Represent error detection time
- Be generated by the component that detects the error
- Be preserved during propagation

MUST NOT

- Be overwritten during propagation
- Be used as the only ordering mechanism in distributed execution

Error Source

Error Source

MUST identify the component that produced the error.

Sources

- Protocol Plugin
- Protocol Runtime
- SIR Runtime
- Planning
- Capability Registry
- Data Kernel
- Engine Runtime
- Engine Plugin
- Session Runtime
- Transaction Runtime
- Control Plane
- Downstream Engine

Error Codes

Error Code

MUST

- Be stable
- Be unique
- Be machine readable
- Use lowercase dot notation
- Belong to one category

Error Code

MUST NOT

- Contain runtime values
- Contain engine-specific native codes directly
- Be reused for different meanings
