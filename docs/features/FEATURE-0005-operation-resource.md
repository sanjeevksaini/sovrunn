# FEATURE-0005 Operation Resource

## 1. Objective

Implement `Operation` as the normalized lifecycle record for platform actions.

Operations provide traceability, auditability, failure explanation, and future workflow integration.

## 2. Resource

```json
{
  "apiVersion": "platform.sovrunn.io/v1alpha1",
  "kind": "Operation",
  "metadata": {
    "name": "op-000001"
  },
  "spec": {
    "operationType": "CreateTenant",
    "targetKind": "Tenant",
    "targetName": "national-health-mission",
    "requestedBy": "system"
  },
  "status": {
    "phase": "Succeeded",
    "message": "Tenant created",
    "startedAt": "2026-07-11T00:00:00Z",
    "completedAt": "2026-07-11T00:00:00Z"
  }
}
```

## 3. Operation Phases

```text
Pending
Running
Succeeded
Failed
Cancelled
```

## 4. API

| Method | Path | Behavior |
|---|---|---|
| GET | `/v1/operations` | List operations |
| GET | `/v1/operations/{name}` | Read operation |
| GET | `/v1/operations?targetKind=Tenant&targetName=national-health-mission` | Filter operations |

In Phase 1, operations are created by internal handlers, not directly by users.

## 5. Validation

- Operation name generated by server.
- Operation type required.
- Target kind/name required.
- Terminal phases are immutable.
- Message should be human-readable.

## 6. Integration Rule

After this feature, all mutating resource APIs must create an Operation record.

For Phase 1, operations can complete synchronously.

## 7. Acceptance Criteria

- Mutating API calls create operation records.
- Can list all operations.
- Can filter by target.
- Failed validations may optionally create failed operation records.
- Tests cover phase transitions and filtering.
