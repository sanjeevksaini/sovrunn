# Requirements Document

## Introduction

FEATURE-0005 introduces the `Operation` resource as the Phase 1 lifecycle/audit record for
API-driven control-plane actions in Sovrunn. An Operation records a single lifecycle action
(create, update, or delete) performed against a governance resource (Organization,
OrganizationUnit, Tenant, Project).

Operation is a platform control-plane resource. In Phase 1 it is emitted **synchronously**
after a successful mutating action and stored in an in-memory registry. It is NOT a workflow
engine: there is no async execution, queueing, retries, step engine, or approval flow.

Existing handlers already contain markers of the form:

```
// TODO(FEATURE-0005): emit Operation record — type: CreateProject
```

FEATURE-0005 replaces those markers with synchronous Operation emission.

This feature depends on FEATURE-0001 through FEATURE-0004 being implemented.

## Glossary

- **Operation**: An immutable control-plane record of a single lifecycle action against a governance resource.
- **Operation ID**: Server-generated unique identifier stored as `metadata.name`.
- **Operation Type**: The action recorded, e.g. `CreateOrganization`, `UpdateTenant`, `DeleteProject`.
- **Resource Reference**: Fields identifying the target resource (kind, name, and parent path where applicable).
- **OperationRegistry**: In-memory, thread-safe store for Operation records, consistent with existing registries.
- **Actor**: The identity that initiated the action. In Phase 1 always `"system"`.
- **Emission**: The act of recording an Operation after a successful mutating action.
- **Error Code**: Stable string from: `VALIDATION_FAILED`, `RESOURCE_NOT_FOUND`, `INTERNAL_ERROR` (and existing codes where applicable).

## Requirements

---

### Requirement 1: Operation Resource Shape

**User Story:** As a platform operator, I want the Operation resource to follow the canonical `metadata/spec/status` shape, so that it is consistent with all Phase 1 resources.

#### Acceptance Criteria

1. THE `Operation` struct SHALL include `APIVersion`, `Kind`, `Metadata`, `Spec`, and `Status` fields with JSON tags `apiVersion`, `kind`, `metadata`, `spec`, and `status` respectively.
2. THE Server SHALL set `apiVersion` to `"platform.sovrunn.io/v1alpha1"` and `kind` to `"Operation"` on all Operation responses.
3. THE `Metadata.Name` field SHALL hold the server-generated Operation ID.
4. THE `OperationSpec` struct SHALL include `Type` (`json:"type"`), `ResourceKind` (`json:"resourceKind"`), `ResourceName` (`json:"resourceName"`), `OrganizationName` (`json:"organizationName,omitempty"`), `OrganizationUnitName` (`json:"organizationUnitName,omitempty"`), `TenantName` (`json:"tenantName,omitempty"`), `ProjectName` (`json:"projectName,omitempty"`), `Actor` (`json:"actor"`), and `RequestID` (`json:"requestId,omitempty"`).
5. THE `OperationStatus` struct SHALL include `Phase` (`json:"phase"`), `Message` (`json:"message,omitempty"`), `CreatedAt` (`json:"createdAt"`), `UpdatedAt` (`json:"updatedAt"`), and `CompletedAt` (`json:"completedAt,omitempty"`).
6. THE `OperationStatus.Phase` SHALL be one of `"Pending"`, `"Running"`, `"Succeeded"`, `"Failed"`.
7. THE Operation resource SHALL NOT contain raw request bodies, secrets, tokens, credentials, or authorization header values.

---

### Requirement 2: Operation Type Enumeration

**User Story:** As a platform operator, I want a fixed set of operation types, so that recorded actions are consistent and queryable.

#### Acceptance Criteria

1. THE `OperationSpec.Type` field SHALL be one of the following stable values: `CreateOrganization`, `UpdateOrganization`, `DeleteOrganization`, `CreateOrganizationUnit`, `UpdateOrganizationUnit`, `DeleteOrganizationUnit`, `CreateTenant`, `UpdateTenant`, `DeleteTenant`, `CreateProject`, `UpdateProject`, `DeleteProject`.
2. THE operation type constants SHALL be defined in `internal/resources` alongside the Operation struct.
3. THE `ResourceKind` field SHALL be one of `Organization`, `OrganizationUnit`, `Tenant`, `Project`.

---

### Requirement 3: Server-Generated Operation IDs

**User Story:** As a platform operator, I want operation IDs generated by the server, so that each recorded action is uniquely identifiable.

#### Acceptance Criteria

1. THE Server SHALL generate the Operation ID; clients SHALL NOT supply it.
2. THE Operation ID SHALL be unique within the OperationRegistry.
3. THE Operation ID SHALL be a URL-safe string suitable for use as a path segment (e.g. a hex or UUID-style token generated via `crypto/rand`).
4. THE Operation ID SHALL be stored in `metadata.name`.

---

### Requirement 4: In-Memory Thread-Safe Operation Registry

**User Story:** As a developer, I want a thread-safe in-memory Operation registry consistent with existing registries.

#### Acceptance Criteria

1. THE OperationRegistry SHALL store Operations in a `map[string]resources.Operation` protected by `sync.RWMutex`, keyed by Operation ID.
2. THE OperationRegistry SHALL use RLock for Get/List and Lock for Create.
3. THE OperationRegistry SHALL return deep copies on Create/Get/List (including any maps or slices).
4. THE OperationRegistry SHALL be storage-only and SHALL NOT generate Operation IDs. `CreateOperation(ctx, op) (resources.Operation, error)` SHALL require `metadata.name` (the Operation ID) to already be set on the submitted Operation. IF `metadata.name` is empty, THEN `CreateOperation` SHALL return a validation/conflict error and SHALL NOT store the Operation.
5. WHEN `CreateOperation` is called with a `metadata.name` that already exists in the registry, THE OperationRegistry SHALL return `ErrAlreadyExists` (reusing the existing conflict sentinel) and SHALL NOT overwrite the existing entry. On success it SHALL store and return the stored deep copy.
6. `GetOperation(ctx, id) (resources.Operation, error)` SHALL return `ErrNotFound` when the ID is absent.
7. `ListOperations(ctx) ([]resources.Operation, error)` SHALL return a non-nil slice (empty when none stored).
8. THE OperationRegistry SHALL accept `context.Context` as the first parameter on all public methods.
9. THE OperationRegistry SHALL be instantiable via constructor; no package-level global state.
10. THE OperationRegistry SHALL produce no data race reports under `go test -race` with 10+ concurrent goroutines.
11. THE OperationRegistry SHALL return results sorted deterministically; List SHALL be sorted by `status.createdAt` ascending, then Operation ID ascending as a tiebreaker.
12. THE OperationRegistry SHALL reuse existing sentinel errors (`ErrNotFound`, `ErrAlreadyExists`) and SHALL remain storage-only with no dependency on other registries.

---

### Requirement 5: Operation Emission from Existing Handlers

**User Story:** As a platform operator, I want lifecycle actions recorded automatically, so that create/update/delete actions leave an audit trail.

#### Acceptance Criteria

1. THE Create, Update, and Delete handlers for Organization, OrganizationUnit, Tenant, and Project SHALL emit an Operation synchronously AFTER the mutating registry action succeeds, at the point where the existing `// TODO(FEATURE-0005)` markers appear.
2. THE emitted Operation SHALL set `spec.type` to the matching operation type, populate the resource reference fields, set `spec.actor = "system"`, and set `spec.requestId` from the request context when available.
3. WHEN emission occurs after a successful synchronous action, THE Operation SHALL be recorded with `status.phase = "Succeeded"` and `status.completedAt` set.
4. THE emission SHALL set `status.createdAt` and `status.updatedAt` to the emission time.
5. THE emission SHALL NOT include raw request bodies, secrets, or credential values.
6. THE existing `// TODO(FEATURE-0005)` marker comments SHALL be replaced by actual emission calls.
7. AFTER implementation, no comment matching `TODO(FEATURE-0005): emit Operation record` SHALL remain anywhere in the source tree. A repository-wide search (e.g. `grep -r "TODO(FEATURE-0005): emit Operation record"`) SHALL return zero matches as an implementation acceptance check.

---

### Requirement 6: Operation Emission Failure Behavior

**User Story:** As a platform operator, I want the main resource action to succeed even if Operation recording fails, so that audit recording never blocks core control-plane behavior in Phase 1.

#### Acceptance Criteria

1. IF Operation emission fails after a successful mutating action, THEN THE Server SHALL NOT fail or reverse the main resource action; the original HTTP response (201/200/204) SHALL be returned as if emission succeeded.
2. WHEN Operation emission fails, THE Server SHALL log a safe warning including the operation type, resource reference, and request ID; the log SHALL NOT include secrets or raw bodies.
3. Emission errors SHALL NOT affect the primary handler response. The design MAY implement a small safe wrapper around emission, but complex panic-recovery machinery is NOT required in Phase 1; a straightforward error-return-and-log approach is acceptable.
4. Emission failures SHALL NOT be surfaced to the client as an error response.

---

### Requirement 7: List Operations — GET /v1/operations

**User Story:** As a platform operator, I want to list recorded Operations.

#### Acceptance Criteria

1. WHEN `GET /v1/operations` is requested, THE Server SHALL return HTTP 200 with `{"items": [...]}`.
2. WHEN no Operations are stored, THE Server SHALL return `{"items": []}` (non-nil empty slice).
3. THE Server SHALL return Operations sorted by `status.createdAt` ascending, then Operation ID ascending.
4. THE response top-level object SHALL contain only the `items` field.
5. On internal error, THE Server SHALL return HTTP 500 with `error.code = "INTERNAL_ERROR"`.

---

### Requirement 8: Get Operation — GET /v1/operations/{name}

**User Story:** As a platform operator, I want to retrieve a single Operation by its ID.

#### Acceptance Criteria

1. WHEN `GET /v1/operations/{name}` is requested and the Operation exists, THE Server SHALL return HTTP 200 with the full Operation resource shape.
2. WHEN the Operation does not exist, THE Server SHALL return HTTP 404 with `error.code = "RESOURCE_NOT_FOUND"`.
3. THE Server SHALL return a deep copy; mutations by the caller SHALL NOT affect registry state.
4. THE Server SHALL use Go 1.21-compatible routing: register `/v1/operations` and `/v1/operations/` patterns, extracting the single `{name}` segment via `strings.TrimPrefix`. A bare `/v1/operations/` with no name SHALL return HTTP 404.

---

### Requirement 9: Operation Creation via API is Out of Scope

**User Story:** As an architect, I want Operations to be system-generated only, so that clients cannot forge audit records in Phase 1.

#### Acceptance Criteria

1. THE Server SHALL NOT expose a client-facing `POST /v1/operations` endpoint in Phase 1.
2. IF a `POST /v1/operations` request is received, THEN THE Server SHALL return HTTP 405 Method Not Allowed.
3. Operations SHALL be created only via internal synchronous emission from mutating handlers (Requirement 5).

---

### Requirement 10: Status Fields Suitable for Future Async Workflows

**User Story:** As an architect, I want Operation status fields shaped for future async workflows, so that FEATURE-0005 does not need refactoring when async execution is added later.

#### Acceptance Criteria

1. THE `OperationStatus.Phase` field SHALL support all four values (`Pending`, `Running`, `Succeeded`, `Failed`) even though Phase 1 only emits `Succeeded` (and `Failed` where a future path needs it).
2. THE `OperationStatus.CompletedAt` field SHALL be optional and set only when the Operation reaches a terminal phase (`Succeeded` or `Failed`).
3. THE Operation shape SHALL NOT include workflow steps, queues, retries, or execution state in Phase 1.
4. THE design SHALL NOT introduce async execution, background goroutines for workflow, or step engines.

---

### Requirement 11: Request Context Usage

**User Story:** As a developer, I want registry methods to accept `context.Context`.

#### Acceptance Criteria

1. THE OperationRegistry methods SHALL each accept `context.Context` as their first parameter.
2. THE Server SHALL pass `r.Context()` to all Operation registry calls within HTTP handlers.
3. Emission from mutating handlers SHALL use the request context (`r.Context()`), not `context.Background()` or `context.TODO()`.
4. THE Server SHALL NOT store `context.Context` in any struct field.

---

### Requirement 12: Server and Wiring

**User Story:** As a developer, I want Operation routes and the registry wired consistently with existing features.

#### Acceptance Criteria

1. THE OperationRegistry SHALL be created in `main.go` and injected into the mutating handlers (OrgHandler, OUHandler, TenantHandler, ProjectHandler) so they can emit Operations, and into an OperationHandler for read endpoints.
2. THE `server.New` signature SHALL be extended to accept a non-nil `*api.OperationHandler`, and SHALL register `/v1/operations` and `/v1/operations/` routes.
3. THE middleware order SHALL remain `requestID → logging → contentType → handler`.
4. Emission dependency SHALL be injected via a narrow emitter interface (e.g. `OperationEmitter`) so handlers do not depend on the concrete registry type; the emitter MAY be nil-safe so existing handler tests continue to pass without an emitter.

---

### Requirement 13: Tests

**User Story:** As a developer, I want comprehensive tests for the Operation resource.

#### Acceptance Criteria

1. THE validation package (if Operation validation is needed) SHALL include unit tests for operation type and resource kind validity.
2. THE registry package SHALL include unit tests: Create stores; Get by ID; Get non-existent → ErrNotFound; List sorted by createdAt then ID; empty List → non-nil empty slice; deep-copy immutability.
3. THE registry package SHALL include a race test with 10+ goroutines performing mixed Create/Get/List.
4. THE api package SHALL include handler tests: GET list 200 sorted/empty; GET 200/404; bare `/v1/operations/` → 404; POST → 405.
5. THE emission tests SHALL verify that a successful create/update/delete on each resource records an Operation with the correct type, resource reference, `actor = "system"`, and `status.phase = "Succeeded"`. These tests MAY be table-driven across the Organization, OrganizationUnit, Tenant, and Project create/update/delete paths to avoid duplicated test code.
6. THE emission tests SHALL verify that a simulated emission failure does NOT change the primary handler response.
7. `go test -race ./...` SHALL produce no race reports.
8. ALL tests SHALL be deterministic with no external dependencies.

---

### Requirement 14: Security and Privacy

**User Story:** As a security-conscious operator, I want Operation records to never store sensitive data.

#### Acceptance Criteria

1. THE Operation SHALL NOT store raw request bodies.
2. THE Operation SHALL NOT store secrets, tokens, credentials, passwords, or authorization header values.
3. Any input summary or parameters recorded SHALL be limited to non-sensitive resource reference fields (kind, name, parent path) and operation type.
4. THE Server SHALL NOT log secrets or raw bodies when recording or reporting Operation emission failures.
5. THE `spec.requestId`, when present, SHALL be the request tracing ID only and SHALL NOT encode sensitive data.

---

### Requirement 15: Edge Cases

**User Story:** As a developer, I want edge-case behavior defined for Operations.

#### Acceptance Criteria

1. WHEN the Operation ID generator produces a value that already exists (collision), THE registry SHALL return an error and the emitter SHALL retry with a new ID a bounded number of times; persistent failure SHALL be treated as an emission failure per Requirement 6.
2. WHEN a mutating action fails (e.g. validation error, duplicate, not found), THE Server SHALL NOT emit an Operation.
3. WHEN `GET /v1/operations/{name}` receives an empty or malformed name segment, THE Server SHALL return HTTP 404 (bare path) — Operation IDs are opaque server-generated tokens and are not DNS-label validated.
4. WHEN the OperationRegistry is empty, `GET /v1/operations` SHALL return `{"items": []}`.
5. WHEN emission is attempted with a nil emitter (e.g. in isolated handler tests), THE handler SHALL skip emission gracefully without panicking.

---

### Requirement 16: Non-Goals

**User Story:** As an architect, I want clear scope boundaries for FEATURE-0005.

#### Acceptance Criteria

1. THE implementation SHALL NOT introduce an async workflow engine, background queue, retries (beyond the bounded ID-collision retry), or a step execution engine.
2. THE implementation SHALL NOT introduce approval workflows, ServiceOps execution, or GitOps integration.
3. THE implementation SHALL NOT introduce an observability pipeline, persistent database storage, or Kubernetes CRDs/controllers.
4. THE implementation SHALL NOT introduce authentication, authorization, or RBAC.
5. THE implementation SHALL NOT introduce AI automation, billing, or UI.
6. THE implementation SHALL NOT introduce distributed locking or multi-node coordination.
7. THE implementation SHALL NOT expose a client-facing Operation creation endpoint.
8. THE implementation SHALL NOT introduce Go 1.22 wildcard routing or new external dependencies.

---

## Compatibility with Existing TODO Markers

The existing handler markers of the form
`// TODO(FEATURE-0005): emit Operation record — type: <Type>`
in the Organization, OrganizationUnit, Tenant, and Project handlers are the exact insertion
points for synchronous Operation emission. FEATURE-0005 replaces each marker with an emission
call using the matching operation type. No marker SHALL remain after implementation.

## Design Questions

> These questions are OPEN and MUST be resolved in `design.md` before implementation begins.

1. **Emitter interface placement:** Should the `OperationEmitter` interface live in `internal/api`
   (consumed by handlers) or `internal/registry` (implemented by the registry)? Recommendation:
   define a narrow emitter interface in `internal/api` that the OperationRegistry satisfies via a
   thin adapter, keeping the registry storage-only.

2. **ID generation location:** Should the Operation ID be generated by the emitter/handler or by
   the registry `CreateOperation`? Recommendation: generate in the emitter (using `crypto/rand`) so
   the registry stays a pure store, with bounded retry on collision.

3. **Phase 1 phase values:** Phase 1 emits only `Succeeded`. Confirm whether any synchronous path
   should ever emit `Failed` in Phase 1, or whether `Failed`/`Pending`/`Running` remain reserved
   for future async workflows.
