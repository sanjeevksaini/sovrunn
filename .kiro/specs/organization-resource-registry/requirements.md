# Requirements Document

## Introduction

FEATURE-0001 implements the `Organization` resource as the top-level governance and ownership boundary in Sovrunn. It also bootstraps the Go module, the minimal project skeleton, the `cmd/sovrunn-api` entrypoint, a minimal HTTP server, and the health/readiness/version endpoints needed for the API server to operate.

Organization is the first resource in the Phase 1 governance hierarchy:

```
Organization
  -> OrganizationUnit (future)
      -> Tenant (future)
          -> Project (future)
```

All resources follow the canonical `metadata/spec/status` shape defined in `docs/resource-specs/RESOURCE_MODEL_PHASE1.md`. The in-memory registry is protected by `sync.RWMutex` and is designed to be replaced by durable storage in a future phase.

This feature covers only `Organization`. It does not implement `OrganizationUnit`, `Tenant`, `Project`, `ServiceClass`, `ServicePlan`, `Plugin`, `Capability`, `ServiceInstance`, `ServiceBinding`, the Operation framework (except a stub boundary), persistent storage, Kubernetes CRDs, GitOps, or any other Phase 1 feature beyond its defined scope.

## Glossary

- **Organization**: Top-level administrative and governance boundary in Sovrunn. Example: `nic` (National Informatics Centre). Defined in `docs/glossary.md`.
- **Registry**: In-memory, thread-safe store for platform resources. Backed by a `sync.RWMutex`-protected map. Replaceable in future phases.
- **Metadata**: Identity and classification fields on a resource: `name`, `displayName`, `labels`, `annotations`.
- **Spec**: Desired state fields of a resource. Set by users or GitOps.
- **Status**: Observed state fields of a resource. System-owned. Not accepted from user input.
- **DNS-label name**: A resource name that matches `^[a-z0-9]([-a-z0-9]*[a-z0-9])?$`. Lowercase, hyphen-separated, no spaces.
- **Request_ID**: Identifier generated by the server (or propagated from `X-Sovrunn-Request-ID` header) for tracing a request. Returned as `X-Sovrunn-Request-ID` in the response.
- **APIError**: Structured JSON error body with `code`, `message`, and optional `field`/`details`.
- **Error_Code**: Stable string error code from the set: `VALIDATION_FAILED`, `RESOURCE_NOT_FOUND`, `RESOURCE_ALREADY_EXISTS`, `DELETE_BLOCKED`, `INTERNAL_ERROR`.
- **sovrunn-api**: The Go binary built from `cmd/sovrunn-api`. Runs the HTTP API server.
- **Server**: The `net/http`-based HTTP server that routes requests to handlers and manages graceful shutdown.
- **Validator**: The package or functions that deterministically validate Organization resource fields.
- **Health_Handler**: HTTP handler for `/healthz`.
- **Readiness_Handler**: HTTP handler for `/readyz`.
- **Version_Handler**: HTTP handler for `/version`.
- **ChildBlockerChecker**: An interface injected into the delete handler that checks whether an Organization has any blocking child resources. In Phase 1 it always returns empty. Future registries plug in via this interface.

### Requirement 1: Go Module and Project Skeleton

**User Story:** As a developer, I want a valid Go module and minimal project skeleton, so that the project compiles and all standard Makefile targets (`make fmt`, `make vet`, `make test`, `make build`) succeed from a clean state.

#### Acceptance Criteria

1. THE Go_Module SHALL be initialized with `go.mod` at the repository root defining the module path `github.com/sanjeevksaini/sovrunn` and Go version 1.21 as the minimum required version.
2. IF the skeleton stage contains only the entrypoint package, THEN THE Go_Module SHALL declare no external module `require` entries beyond the Go standard library.
3. WHEN `make fmt` is executed, THE Go_Module SHALL produce zero formatting changes across all Go source files.
4. WHEN `make vet` is executed, THE Go_Module SHALL produce zero vet errors across all Go source files.
5. IF only the entrypoint package exists (skeleton stage), THEN WHEN `make test` is executed, THE Go_Module SHALL complete with exit code 0.
6. WHEN `make build` is executed, THE Go_Module SHALL produce a binary at `bin/sovrunn-api` without error.
7. THE Go_Module SHALL use the standard library `net/http` package for the HTTP server; no third-party HTTP framework SHALL be introduced unless explicitly approved.

### Requirement 2: Minimal Go Project Structure

**User Story:** As a developer, I want a consistent internal package layout, so that all Phase 1 features can be added to well-defined, responsibility-scoped packages without refactoring.

#### Acceptance Criteria

1. THE Project SHALL provide the following package directories: `cmd/sovrunn-api/`, `internal/api/`, `internal/config/`, `internal/health/`, `internal/registry/`, `internal/resources/`, `internal/server/`, `internal/validation/`.
2. THE `cmd/sovrunn-api/` package SHALL contain `main.go` and SHALL be limited to dependency wiring and server startup; business logic SHALL NOT be placed in `main.go`.
3. THE `internal/resources/` package SHALL define the `Organization` struct and all sub-structs (`Metadata`, `OrganizationSpec`, `OrganizationStatus`) using explicit JSON tags per `docs/engineering/go-coding-guardrails.md` section 9.
4. THE `internal/registry/` package SHALL define the `OrganizationRegistry` struct backed by `sync.RWMutex` and a `map[string]resources.Organization`, and SHALL expose `CreateOrganization`, `GetOrganization`, `ListOrganizations`, `UpdateOrganization`, and `DeleteOrganization` as its public API.
5. THE `internal/validation/` package SHALL define `ValidateOrganization` as a pure function with no external module dependencies, returning a `[]FieldError` where `FieldError` carries `Field` (dot-separated JSON path) and `Message` string fields.
6. THE `internal/api/` package SHALL define HTTP handler functions for Organization CRUD endpoints and SHALL NOT contain business logic beyond request decoding, validation invocation, registry invocation, and response encoding.
7. THE `internal/server/` package SHALL own HTTP server lifecycle including startup, route registration, and graceful shutdown with a maximum shutdown timeout of 30 seconds.
8. THE `internal/config/` package SHALL define a `Config` struct loaded from the YAML config file at `configs/sovrunn-api.local.yaml`; IF the config file is missing or contains invalid values, THEN THE server process SHALL exit with a non-zero status code and log a startup error before accepting any requests.

### Requirement 3: API Server Entrypoint and HTTP Server

**User Story:** As a developer, I want a running HTTP API server launched from `cmd/sovrunn-api`, so that the platform API is reachable at the configured host and port.

#### Acceptance Criteria

1. WHEN `make run` is executed, THE Server SHALL start and listen on the host and port specified in `configs/sovrunn-api.local.yaml` (default `127.0.0.1:8080`).
2. THE Server SHALL register the routes `POST /v1/organizations`, `GET /v1/organizations`, `GET /v1/organizations/{name}`, `PUT /v1/organizations/{name}`, `DELETE /v1/organizations/{name}`, `GET /healthz`, `GET /readyz`, and `GET /version` at startup before accepting requests.
3. WHEN the Server receives `SIGINT` or `SIGTERM`, THE Server SHALL stop accepting new requests, allow in-flight requests to complete within a 30-second shutdown timeout, and exit with code 0.
4. THE Server SHALL set `Content-Type: application/json` on all JSON responses.
5. IF the `X-Sovrunn-Request-ID` request header is absent or is an empty string, THEN THE Server SHALL generate a unique `request_id` for the request.
6. WHEN the `X-Sovrunn-Request-ID` header is present and non-empty on an incoming request, THE Server SHALL use that value as the request ID without modification.
7. THE Server SHALL include the resolved `request_id` in the `X-Sovrunn-Request-ID` response header on every response.

### Requirement 4: Health, Readiness, and Version Endpoints

**User Story:** As an operator, I want `/healthz`, `/readyz`, and `/version` endpoints, so that the API server liveness, readiness, and build information can be verified without authentication.

#### Acceptance Criteria

1. WHEN `GET /healthz` is requested, THE Health_Handler SHALL return HTTP 200 with body `{"status":"ok"}`.
2. WHEN `GET /readyz` is requested and the Server has successfully bound its HTTP listener, initialized the registry, loaded its config, and encountered no fatal init error, THE Readiness_Handler SHALL return HTTP 200 with body `{"status":"ready"}`.
3. WHEN `GET /readyz` is requested and the Server has not yet completed initialization or has encountered a fatal init error, THE Readiness_Handler SHALL return HTTP 503 with body `{"status":"not-ready"}`.
4. WHEN `GET /version` is requested, THE Version_Handler SHALL return HTTP 200 with a JSON body containing `name` (string), `version` (string), `phase` (string), and `status` (string) fields.
5. THE Health_Handler SHALL NOT call the registry, perform I/O, or execute any blocking operation.
6. THE Readiness_Handler SHALL NOT call external services or perform blocking I/O in Phase 1.
7. THE `/healthz`, `/readyz`, and `/version` endpoints SHALL NOT require any authentication header to return a response.

### Requirement 5: Organization Resource Shape

**User Story:** As a platform operator, I want the Organization resource to follow the canonical `metadata/spec/status` shape, so that it is consistent with all future Phase 1 resources and can be evolved toward Kubernetes-compatible desired-state reconciliation.

#### Acceptance Criteria

1. THE `Organization` struct SHALL include `APIVersion`, `Kind`, `Metadata`, `Spec`, and `Status` fields with JSON tags `apiVersion`, `kind`, `metadata`, `spec`, and `status` respectively.
2. THE `Metadata` struct SHALL use JSON tag `json:"name"` (no omitempty) for `Name`, and `json:"displayName,omitempty"`, `json:"labels,omitempty"`, `json:"annotations,omitempty"` for the optional fields.
3. THE `OrganizationSpec` struct SHALL include `Description` (`json:"description,omitempty"`), `SovereignLocations` (`json:"sovereignLocations,omitempty"`, type `[]string`), and `DefaultPolicyProfile` (`json:"defaultPolicyProfile,omitempty"`).
4. THE `OrganizationStatus` struct SHALL include `Phase` (`json:"phase"`) with valid values `"Active"`, `"Inactive"`, `"Deleting"`, `"Failed"`, and `Message` (`json:"message,omitempty"`).
5. WHEN the Server returns a successful Organization response (HTTP 200 or 201), THE Server SHALL set `apiVersion` to `"platform.sovrunn.io/v1alpha1"` and `kind` to `"Organization"` regardless of what the client submitted.
6. IF the top-level JSON request body on a POST or PUT request contains the key `status` — regardless of value (`{}`, `null`, `{"phase": ""}`, or any other value) — THEN THE Server SHALL return HTTP 400 with `error.code = "VALIDATION_FAILED"` and `error.field = "status"`.
7. WHEN `POST /v1/organizations` succeeds, THE Server SHALL set `status.phase` to `"Active"` both in the stored registry entry and in the 201 response body.

### Requirement 6: Organization Name Validation

**User Story:** As a platform operator, I want deterministic Organization name validation, so that only safe, consistent, and DNS-compatible names are accepted and stored.

#### Acceptance Criteria

1. IF `metadata.name` is absent or an empty string, THEN THE Validator SHALL return a `FieldError` with `Field = "metadata.name"` and skip further name validation rules.
2. IF `metadata.name` is non-empty and does not match the pattern `^[a-z0-9]([-a-z0-9]*[a-z0-9])?$`, THEN THE Validator SHALL return a `FieldError` with `Field = "metadata.name"` indicating an invalid DNS-label format.
3. IF `metadata.name` is non-empty and its length exceeds 63 characters, THEN THE Validator SHALL return a `FieldError` with `Field = "metadata.name"` indicating the name is too long.
4. IF `metadata.name` is non-empty and matches `^[a-z0-9]([-a-z0-9]*[a-z0-9])?$` and has length between 1 and 63 inclusive, THEN THE Validator SHALL return a result containing zero validation errors for the name field.
5. THE Validator SHALL be callable as a pure function without starting the HTTP server or instantiating the registry.
6. THE Validator SHALL return `[]FieldError` to support multiple field errors in future phases. In FEATURE-0001 scope, only `metadata.name` has validation rules, so at most one `FieldError` will be returned per call. The `[]FieldError` return type is retained for forward compatibility.

### Requirement 7: In-Memory Thread-Safe Organization Registry

**User Story:** As a developer, I want a thread-safe in-memory Organization registry, so that concurrent API requests can safely read and write Organization state without data races.

#### Acceptance Criteria

1. THE Registry SHALL store Organizations in a `map[string]resources.Organization` protected by a `sync.RWMutex`.
2. THE Registry SHALL use a read lock (`RLock`/`RUnlock`) for `GetOrganization` and `ListOrganizations` operations.
3. THE Registry SHALL use a write lock (`Lock`/`Unlock`) for `CreateOrganization`, `UpdateOrganization`, and `DeleteOrganization` operations.
4. THE Registry SHALL NOT return a reference to its internal map; `GetOrganization` SHALL return a deep copy of the stored `Organization` struct, including independent copies of `Metadata.Labels`, `Metadata.Annotations` maps and `Spec.SovereignLocations` slice.
5. WHEN `ListOrganizations` is called, THE Registry SHALL return a new `[]resources.Organization` slice whose elements are deep copies of stored Organizations (including independent copies of maps and slices).
6. THE Registry SHALL accept `context.Context` as the first parameter on all five public methods.
7. THE Registry SHALL be instantiable as a zero-value struct or via a constructor; no package-level global variables SHALL hold registry state.
8. WHEN run under `go test -race ./...` with at least 10 concurrent goroutines performing mixed reads and writes, THE Registry SHALL produce no data race reports.
9. THE Registry SHALL return a typed error value (or sentinel error) distinguishing `RESOURCE_NOT_FOUND` from `RESOURCE_ALREADY_EXISTS` so that HTTP handlers can map registry errors to the correct APIError code without inspecting error message strings.

### Requirement 8: Create Organization — POST /v1/organizations

**User Story:** As a platform operator, I want to create an Organization via REST API, so that I can register a new top-level governance boundary in Sovrunn.

#### Acceptance Criteria

1. WHEN a valid `POST /v1/organizations` request is received, THE Server SHALL validate the request body, store the Organization in the Registry, and return HTTP 201 with a JSON body containing the full Organization resource including server-set `apiVersion`, `kind`, and `status.phase = "Active"`.
2. WHEN the request body contains a `metadata.name` that already exists in the Registry, THE Server SHALL return HTTP 409 with `error.code = "RESOURCE_ALREADY_EXISTS"`.
3. WHEN the request body fails validation, THE Server SHALL return HTTP 400 with `error.code = "VALIDATION_FAILED"`, `error.message` describing the failure, and `error.field` set to the dot-separated JSON path of the first invalid field.
4. WHEN the request body is not valid JSON, THE Server SHALL return HTTP 400 with `error.code = "VALIDATION_FAILED"` and `error.message` indicating malformed JSON.
5. WHEN the request `Content-Type` is not `application/json`, THE Server SHALL return HTTP 415 with `error.code = "VALIDATION_FAILED"`.
6. WHEN `POST /v1/organizations` succeeds, THE Server SHALL always set `status.phase = "Active"` regardless of any `status` value in the request body.
7. IF the top-level JSON request body contains the key `status` — regardless of value (`{}`, `null`, `{"phase": ""}`, or any other value) — THEN THE Server SHALL return HTTP 400 with `error.code = "VALIDATION_FAILED"` and `error.field = "status"` without storing the resource.
8. WHEN the Organization is successfully created, THE Server SHALL store a deep copy (including independent copies of maps and slices) such that a subsequent `GET /v1/organizations/{name}` with the same name returns HTTP 200 with the stored resource.

---

### Requirement 9: Get Organization — GET /v1/organizations/{name}

**User Story:** As a platform operator, I want to retrieve a single Organization by name, so that I can inspect its current metadata, spec, and status.

#### Acceptance Criteria

1. WHEN `GET /v1/organizations/{name}` is requested and the Organization exists, THE Server SHALL return HTTP 200 with a JSON body containing the full Organization resource shape (`apiVersion`, `kind`, `metadata`, `spec`, `status`).
2. WHEN `GET /v1/organizations/{name}` is requested and the Organization does not exist, THE Server SHALL return HTTP 404 with `error.code = "RESOURCE_NOT_FOUND"`.
3. WHEN `GET /v1/organizations/{name}` is requested, THE Server SHALL use the value of `{name}` extracted from the URL path as the registry lookup key.
4. IF the `{name}` path segment does not match the DNS-label pattern `^[a-z0-9]([-a-z0-9]*[a-z0-9])?$` or exceeds 63 characters, THEN THE Server SHALL return HTTP 400 with `error.code = "VALIDATION_FAILED"` without performing a registry lookup.
5. THE Server SHALL return a deep copy of the Organization (including independent copies of `Metadata.Labels`, `Metadata.Annotations` maps and `Spec.SovereignLocations` slice); mutations to the returned struct by the caller SHALL NOT affect the state stored in the Registry.

---

### Requirement 10: List Organizations — GET /v1/organizations

**User Story:** As a platform operator, I want to list all Organizations, so that I can see all registered top-level governance boundaries.

#### Acceptance Criteria

1. WHEN `GET /v1/organizations` is requested, THE Server SHALL return HTTP 200 with a JSON body containing an `items` array of all stored Organizations.
2. WHEN no Organizations are stored, THE Server SHALL return HTTP 200 with `{"items": []}`.
3. WHEN `GET /v1/organizations` is requested, THE Server SHALL return Organizations sorted in ascending lexicographic order of `metadata.name`.
4. THE response body top-level object SHALL contain only the `items` array field; no additional registry-internal fields SHALL appear at the top level.
5. WHEN the registry encounters an unexpected internal error during list, THE Server SHALL return HTTP 500 with `error.code = "INTERNAL_ERROR"`.

---

### Requirement 11: Update Organization — PUT /v1/organizations/{name}

**User Story:** As a platform operator, I want to update mutable fields of an Organization, so that I can change its description, display name, labels, and sovereign locations over time.

#### Acceptance Criteria

1. WHEN a valid `PUT /v1/organizations/{name}` request is received and the Organization exists, THE Server SHALL replace `metadata.displayName`, `metadata.labels`, `metadata.annotations`, `spec.description`, `spec.sovereignLocations`, and `spec.defaultPolicyProfile` with the values from the request body, preserve `metadata.name` and all `status` fields from the stored entry unchanged, and return HTTP 200 with the updated Organization as a JSON body.
2. WHEN `PUT /v1/organizations/{name}` is requested and the Organization does not exist, THE Server SHALL return HTTP 404 with `error.code = "RESOURCE_NOT_FOUND"`.
3. THE Server SHALL allow updates to `metadata.displayName`, `metadata.labels`, `metadata.annotations`, `spec.description`, `spec.sovereignLocations`, and `spec.defaultPolicyProfile`.
4. THE Server SHALL require `metadata.name` to be present and non-empty in the PUT request body. IF `metadata.name` is absent or empty, THEN THE Server SHALL return HTTP 400 with `error.code = "VALIDATION_FAILED"` and `error.field = "metadata.name"`. IF `metadata.name` is present but differs from the `{name}` path segment, THEN THE Server SHALL return HTTP 400 with `error.code = "VALIDATION_FAILED"` and `error.field = "metadata.name"`.
5. IF the top-level JSON request body contains the key `status` — regardless of value (`{}`, `null`, `{"phase": ""}`, or any other value) — THEN THE Server SHALL return HTTP 400 with `error.code = "VALIDATION_FAILED"` and `error.field = "status"`.
6. WHEN the request body is not valid JSON, THE Server SHALL return HTTP 400 with `error.code = "VALIDATION_FAILED"`.
7. WHEN the `Content-Type` header is not `application/json`, THE Server SHALL return HTTP 415 with `error.code = "VALIDATION_FAILED"`.

---

### Requirement 12: Delete Organization — DELETE /v1/organizations/{name}

**User Story:** As a platform operator, I want to delete an Organization only when it has no child resources, so that governance boundaries are not accidentally removed while resources depend on them.

#### Acceptance Criteria

1. WHEN `DELETE /v1/organizations/{name}` is requested, the Organization exists, and the child-resource blocker check returns no blocking resources, THE Server SHALL remove the Organization from the Registry and return HTTP 204 with no body.
2. WHEN `DELETE /v1/organizations/{name}` is requested and the Organization does not exist, THE Server SHALL return HTTP 404 with `error.code = "RESOURCE_NOT_FOUND"`.
3. WHEN `DELETE /v1/organizations/{name}` is requested and the child-resource blocker check returns one or more blocking resource kinds, THE Server SHALL return HTTP 409 with `error.code = "DELETE_BLOCKED"` and `error.message` naming the blocking resource kind.
4. In Phase 1 (FEATURE-0001 scope), the child-resource blocker check SHALL always return an empty blocking set, so HTTP 204 is the only successful outcome for an existing Organization.
5. THE delete handler SHALL invoke the child-resource blocker check through a `ChildBlockerChecker` interface so that a stub implementation can be injected in tests and future registries (OrganizationUnit, Tenant, Project) can be plugged in without modifying the handler.

---

### Requirement 13: Structured JSON Error Responses

**User Story:** As an API consumer, I want all errors to be returned as structured JSON with stable error codes, so that I can programmatically handle and display failures.

#### Acceptance Criteria

1. THE Server SHALL return all error responses as JSON with the shape `{"error": {"code": "<Error_Code>", "message": "<string>", "field": "<optional string>", "details": "<optional string>"}}`.
2. THE Server SHALL use only the stable Error_Code values: `VALIDATION_FAILED`, `RESOURCE_NOT_FOUND`, `RESOURCE_ALREADY_EXISTS`, `DELETE_BLOCKED`, `INTERNAL_ERROR`.
3. THE Server SHALL map error codes to HTTP status codes as follows: `VALIDATION_FAILED` → 400, `RESOURCE_NOT_FOUND` → 404, `RESOURCE_ALREADY_EXISTS` → 409, `DELETE_BLOCKED` → 409, `INTERNAL_ERROR` → 500, content-type mismatch → 415, oversized body → 413.
4. THE Server SHALL NOT include Go stack traces, internal package paths, or internal variable names in any error response.
5. THE Server SHALL NOT log secret values, authorization header contents, tokens, or private keys when handling or recording errors.
6. WHEN a validation error identifies a specific invalid field, THE Server SHALL populate `error.field` with the dot-separated JSON path of the offending field (e.g., `"metadata.name"`).
7. WHEN a single request produces multiple validation errors across different fields, THE Server SHALL include all field errors; the primary `error.field` SHALL be the first failing field, and `error.details` SHALL list the remaining failing fields.

### Requirement 14: Safe JSON Decoding with Request Body Limit

**User Story:** As a security-conscious operator, I want all JSON decoding to be safe against oversized bodies and unknown fields, so that the API server is protected against basic resource-exhaustion and injection risks.

#### Acceptance Criteria

1. THE Server SHALL apply `http.MaxBytesReader` with a limit of 1 MiB (1,048,576 bytes) to every request body before decoding.
2. THE Server SHALL use `json.Decoder.DisallowUnknownFields()` when decoding request bodies. Unknown fields such as `resourceVersion`, `generation`, `createdAt`, and `updatedAt` SHALL be rejected automatically by the decoder. The top-level `status` key SHALL be rejected explicitly before the typed decode (see Requirement 5.6).
3. IF JSON decoding returns an error because the body exceeds 1 MiB, THE Server SHALL return HTTP 413 with `error.code = "VALIDATION_FAILED"`.
4. IF JSON decoding returns a syntax error, a type-mismatch error, or an unknown-field error, THE Server SHALL return HTTP 400 with `error.code = "VALIDATION_FAILED"` and a message indicating the nature of the problem.
5. THE Server SHALL NOT echo back the raw request body in any error response.

---

### Requirement 15: Request Context Usage

**User Story:** As a developer, I want all registry and validation functions to accept `context.Context`, so that request deadlines, cancellations, and request-scoped values propagate correctly through the call stack.

#### Acceptance Criteria

1. THE `ValidateOrganization` function and the Registry methods `CreateOrganization`, `GetOrganization`, `ListOrganizations`, `UpdateOrganization`, and `DeleteOrganization` SHALL each accept `context.Context` as their first parameter.
2. THE Server SHALL pass `r.Context()` as the context argument to all `ValidateOrganization` and registry method calls within HTTP handlers.
3. THE Server SHALL NOT store `context.Context` in any struct field.
4. THE Server SHALL NOT pass `context.Background()` or `context.TODO()` inside HTTP handlers in place of the request context.

---

### Requirement 16: No Secret Logging

**User Story:** As a security-conscious operator, I want the API server to never log secrets, tokens, or credential values, so that sensitive platform data is never written to log output.

#### Acceptance Criteria

1. THE Server SHALL NOT log the value of the `Authorization` header.
2. THE Server SHALL NOT log full request body payloads by default.
3. THE Server SHALL NOT include secret values, tokens, private keys, or passwords in any log line, structured log field, or error message returned to API clients.
4. THE Server SHALL log the following fields for every request: `request_id`, `method`, `path`, `status_code`, `latency_ms`; and on failure additionally `error_code`.
5. WHEN an internal error occurs, THE Server SHALL log a safe summary message and the `error_code` only; the raw Go error chain and internal struct state SHALL NOT appear in the response body.

### Requirement 18: Tests for Validation, Registry, and HTTP Handlers

**User Story:** As a developer, I want unit tests for Organization validation, the in-memory registry, and the HTTP handlers, so that correctness is proven by the test suite and regressions are caught automatically.

#### Acceptance Criteria

1. THE `internal/validation/` package SHALL include tests that verify: (a) valid names are accepted, (b) empty name is rejected, (c) names with uppercase letters are rejected, (d) names with spaces are rejected, (e) names exceeding 63 characters are rejected, (f) names with leading or trailing hyphens are rejected.
2. THE `internal/registry/` package SHALL include tests that verify: (a) Create stores an Organization, (b) duplicate Create returns `RESOURCE_ALREADY_EXISTS` error, (c) Get returns the stored Organization, (d) Get for a non-existent name returns `RESOURCE_NOT_FOUND` error, (e) List returns all stored Organizations in lexicographic order, (f) List on empty registry returns an empty slice, (g) Update modifies mutable fields, (h) Update of a non-existent Organization returns `RESOURCE_NOT_FOUND` error, (i) Delete removes the Organization, (j) Delete of a non-existent Organization returns `RESOURCE_NOT_FOUND` error.
3. THE `internal/api/` package SHALL include HTTP handler tests using `net/http/httptest` that verify: (a) POST returns 201 for valid input, (b) POST returns 409 for duplicate name, (c) POST returns 400 for invalid name, (d) POST returns 400 for user-authored status field, (e) GET returns 200 for existing resource with full resource shape, (f) GET returns 404 for missing resource, (g) GET list returns 200 with `items` array sorted lexicographically, (h) PUT returns 200 for valid update, (i) PUT returns 404 for missing resource, (j) PUT returns 400 when metadata.name in body differs from path, (k) DELETE returns 204 for existing resource, (l) DELETE returns 404 for missing resource.
4. WHEN `go test -race ./...` is executed, THE test suite SHALL produce no data race reports.
5. ALL tests SHALL be deterministic and SHALL NOT depend on external services, network access, or filesystem state beyond the repository root.

### Requirement 19: Operation Framework Boundary (Placeholder Only)

**User Story:** As an architect, I want the FEATURE-0001 implementation to define a clear operation boundary, so that FEATURE-0005 can wire in Operation records for mutating actions without requiring API handler refactoring.

#### Acceptance Criteria

1. THE `internal/api/` handlers for `CreateOrganization`, `UpdateOrganization`, and `DeleteOrganization` SHALL include a clearly commented placeholder at the point where an Operation record will be created once FEATURE-0005 is implemented.
2. THE placeholder SHALL NOT import the `internal/operation/` package or call any Operation creation code; it SHALL be a code comment only.
3. THE placeholder comment SHALL identify the expected Operation type: `// TODO(FEATURE-0005): emit Operation record — type: CreateOrganization`, `UpdateOrganization`, or `DeleteOrganization` respectively.
4. THE implementation SHALL NOT create an `internal/operation/` package or any Operation structs as part of FEATURE-0001.
